Revelation: Lambda Calculus Reduced To Four Primitive Operations

Darius J Chuck

2023-07-23

Lambda calculus can be reduced down to 4 primitive operations.

First, let‚Äôs define a simplified grammar for a lambda calculus term that captures these operations syntactically:

term ::= abstraction
       | application
       | revelation
       | reference

A term is defined as either of the four basic operations.

We don‚Äôt want to worry about parentheses, associativity, and precedence, so each operation shall use a distinct prefix operator with fixed arity. Effectively we‚Äôll be writing our terms in Polish notation ‚Äì the order of operations explicitly defined by the position of the operators, which are always evaluated left-to-right.

The first two operations should be more or less familiar.

Abstraction is defined like so:

abstraction ::= ùõå\boldsymbol{\lambda} term

The definition doesn‚Äôt mention variable names. We only have a lambda Œª\lambda, which we‚Äôll call the abstraction operator, prefixed to a term. This is the same as in N.G. de Bruijn‚Äôs namefree syntax.

Now application is defined as:

application ::= ùõÇ\boldsymbol{\alpha} term term

Instead of using juxtaposition, we introduce a two-argument application operator, alpha Œ±\alpha. Predictably, it applies the second term to the first.

With that out of the way, let‚Äôs get to the interesting operations: revelation and reference.

As can be guessed, these replace variable references.

We will leave revelation for the end. First let‚Äôs look at reference:

reference ::= ùõï\boldsymbol{\tau}

It‚Äôs denoted simply by the letter tau œÑ\tau, which is our zero-argument reference operator. What it does is it refers to the topmost variable in current scope.

Normally the topmost variable is the one introduced by the lambda closest to the current position. That is:

ŒªœÑ \lambda \tau

means the same thing as Œªx.x\lambda x.x. in the conventional notation. Similarly:

ŒªŒªœÑ \lambda \lambda \tau

means the same thing as Œªx.Œªy.y\lambda x.\lambda y.y in the conventional notation.

That‚Äôs all well and good, but how do we refer to a variable that is not topmost in current scope?

Here is where the revelation comes in:

revelation ::= ùõî\boldsymbol{\sigma} term

Besides being a cool pun, the name is quite descriptive. The revelation operator, denoted by sigma œÉ\sigma, creates a new scope for the term that follows it, with the topmost variable discarded, thus revealing the next variable to the term. The topmost variable in the new scope is the one introduced by the lambda one step back from the current position. So:

ŒªŒªœÉœÑ \lambda \lambda \sigma \tau

means the same thing as Œªx.Œªy.x\lambda x.\lambda y.x in the conventional notation.

With these 4 operations, not only can we encode any and all lambda terms that we can encode in the conventional notation without using numerical de Bruijn indices or needing an infinite supply of names for variables.

Being more ‚Äúlow-level‚Äù, this notation can actually represent most lambda terms in more than one way.

For example, a conventionally-encoded lambda term like:

Œªx.Œªy.Œªz.x \lambda x.\lambda y.\lambda z.x

Can be represented as:

ŒªŒªŒªœÉœÉœÑ \lambda \lambda \lambda \sigma \sigma \tau

or equivalently as:

ŒªŒªœÉŒªœÉœÑ \lambda \lambda \sigma \lambda \sigma \tau

Having œÉ\sigma be an independent first-class operation, we can now choose the points at which we reveal the interesting (or equivalently: discard the uninteresting) variables.

This starts to matter when we add applications to the mix.

Consider a term like:

Œªx.Œªy.Œªz.xxxx \lambda x.\lambda y.\lambda z.x\,x\,x\,x

But actually, we don‚Äôt have to repeat the revelation operation twice for every application. We can ‚Äúfactor it out‚Äù, like so:

ŒªŒªŒªœÉœÉŒ±Œ±Œ±œÑœÑœÑœÑ \lambda \lambda \lambda \sigma \sigma \alpha \alpha \alpha \tau \tau \tau \tau

Now the scopes which the œÑ\tau operations ‚Äúsee‚Äù already have the unnecessary variables discarded.

This equivalent encoding of the lambda term is not only shorter but more efficient in terms of the number of operations performed.

With that, we are ready to introduce the notions of œÉ\sigma-equivalence and œÉ\sigma-reduction or œÉ\sigma-optimization, along with its inverse, œÉ\sigma-deoptimization.

The implementation also features œÉ\sigma-optimization, there called S-optimization, which is an algorithm I developed that transforms lambda terms into fully optimized œÉ\sigma-equivalents.
